# Lead prioritization: решение и описание подхода

## Цель задачи

Нужно предсказать вероятность успешного целевого действия в течение 5 дней после назначения обращения. Итоговый score используется для ранжирования обращений: более перспективные обращения должны получать более высокий score.

## Что было сделано

Решение построено в ноутбуке [quickstart.ipynb](quickstart.ipynb) и состоит из следующих шагов:

1. Загрузка обучающей и тестовой выборки, а также дополнительных событий из [data/events.csv](data/events.csv).
2. Построение признаков, доступных на момент назначения обращения.
3. Обучение модели и оценка качества на временной валидации.
4. Обучение на всех доступных данных и создание файла [submission.csv](submission.csv).

## Как оценивалось качество

Поскольку задача является задачей ранжирования, качество оценивалось с помощью Average Precision (AP). Для более реалистичной оценки использовалась временная валидация:

- ранние даты использовались для обучения;
- поздние даты - для проверки.

Это важнее случайного сплита, потому что в реальной задаче модель должна работать на данных, которые приходят позже по времени

## Почему использовалась именно такая валидация

В задаче есть временная структура:

- обучающие данные идут раньше по времени;
- тестовая выборка приходит позже;
- признаки должны быть доступны на момент назначения обращения.

Поэтому временной split лучше соответствует production-условиям и снижает риск переоценки качества

## Обработка пропусков, категориальных признаков и редких значений

Для подготовки данных использовался стандартный и устойчивый пайплайн:

- числовые признаки:
  - заполнение пропусков медианой;
  - стандартизация.
- категориальные признаки:
  - заполнение пропусков самым частым значением;
  - one-hot encoding.


## Feature engineering

Помимо готовых признаков из train/test, в решение были добавлены признаки из событий из [data/events.csv](data/events.csv).

Признаки строились только из событий, которые произошли до момента назначения обращения, это важно, чтобы не допустить утечку и не использовать информацию, которая в проде ещё не была бы доступна

### Что именно было добавлено

Для каждого обращения и пользователя были посчитаны:

- общее число событий;
- число событий по типам: item_view, search, favorite, chat_open, call_click;
- агрегаты по временным окнам: 1d, 3d, 7d, 14d, 30d;
- средняя/максимальная/минимальная цена товара из событий;
- число уникальных src_slot и ctx_seq;
- близость последнего события ко времени назначения.

## Почему была выбрана именно эта модель

Были сравнены несколько подходов:

- логистическая регрессия;
- случайный лес;
- градиентный бустинг;
- HistGradientBoostingClassifier;
- SVM;
- MLP.

На временной валидации лучший результат показал HistGradientBoostingClassifier.

### Почему она была выбрана

- хорошо работает на табличных данных с нелинейными зависимостями;
- устойчиво справляется со смешанным набором числовых и категориальных признаков;
- даёт хороший AP на валидации без избыточной сложности;
- проще и быстрее в обучении, чем более тяжёлые ансамблевые подходы.

На временной валидации итоговое качество составило около 0.65158 AP, что лучше, чем у базового логистического регрессора.


## Краткое объяснение подхода

Подход можно описать очень коротко так:

- берём признаки, доступные на момент назначения;
- добавляем признаки из событий, но только те, что появились до назначения;
- готовим признаки для модели через стандартный preprocessing;
- обучаем градиентный бустинг на деревьях;
- оцениваем качество на временной валидации и формируем score для теста.
